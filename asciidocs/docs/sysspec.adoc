=== Data Flow Sequence Diagram

The following diagram illustrates the real-time data flow from the sensors to the end-user dashboard.

----
@startuml

== Periodic Data Collection ==

sensors -> mqtt : Publish room data\nTopic: room-temperature\nPayload: {"room": "105", "temperature": 22.5}
mqtt -> telegraf : Forward message
telegraf -> influx : Write to bucket "server_data"\nMeasurement: room_temperature

== Real-time Visualization (WebSocket) ==

user -> web : Opens Dashboard
web -> ws : Connect & Subscribe\n{"type": "subscribe", "room": "105"}
mqtt -> ws : Forward live message
ws -> web : Send live update\n{"type": "temp", "room": "105", "value": 22.5}
web -> user : Update Heatmap / Chart

== Historical Data (InfluxDB API) ==

user -> web : Selects Date Range
web -> influx : Query via Flux API\n/api/v2/query
influx -> web : Return CSV data
web -> user : Render Historical Charts
@enduml
----

=== Component Specification

==== MQTT Topic Hierarchy

The project uses the following MQTT topics for communication:

[cols="1,2,2", options="header"]
|===
| Topic | Format | Description
| `room-temperature` | JSON: `{"room": "ID", "temperature": float}` | Periodic temperature updates for all rooms.
| `nili3/sensor/nili3_co2/state` | Value (Float) | CO2 levels for Room 105 (special sensor).
| `nili3/sensor/nili3_temperature/state` | Value (Float) | High-resolution temperature for Room 105.
| `nili3/sensor/+/state` | Value (Float) | Generic pattern for other Nili sensors.
|===

==== InfluxDB Schema (Bucket: `server_data`)

[cols="1,1,2,2", options="header"]
|===
| Measurement | Field | Tags | Description
| `room_temperature` | `temperature` | `room` | Standardized room temperatures.
| `mqtt_consumer` | `value` | `topic` | Raw data from various MQTT topics (CO2, power, etc).
| `pv_data` | (multiple) | `plant_id` | Solar plant metrics (yield, consumption, battery).
|===

== Functional Requirements

=== Real-time Monitoring
* Users can view the current temperature and CO2 levels of every room in a 3D building model.
* Data is updated instantly via WebSockets when new sensor values arrive.

=== Historical Analysis
* Users can select a specific date or time range (6h, 24h, 7d) to view trends.
* Aggregated averages are shown for rooms and floors.

=== Mobile Accessibility
* The dashboard is fully responsive and optimized for mobile devices (iOS/Android).
* Support for notch/Dynamic Island devices and touch-friendly interaction targets.

== Non-functional Requirements (NFA)
* **Latency:** Real-time updates should reflect in the UI within 500ms of reaching the MQTT broker.
* **Scalability:** The system should handle up to 200 rooms with 10-second update intervals.
* **Availability:** The dashboard and data collectors should restart automatically via Docker Compose.
